# How to Store Passwords in a Database — A Complete Guide

> Based on a security interview discussion. A deep dive into one of the most fundamental questions every backend and security engineer should be able to answer.

---

## Introduction

"How do you store passwords in a database?"

It sounds like a simple question, but it is one of the most common — and most revealing — questions asked in security interviews. The answer touches on cryptography, attack vectors, and real-world engineering trade-offs.

In this article, we will walk through everything you need to know:

- Why we **hash** passwords instead of **encrypting** them
- Why **salt** matters
- What a **rainbow table** attack is
- Which **hashing algorithm** to choose (and which to avoid)
- Why **MD5** and **SHA-1** are dead for passwords
- What a **KDF** (Key Derivation Function) is and why it matters

---

## 1. Plaintext is Never an Option

In the 21st century, storing passwords as plaintext is unforgivable. If your database is ever leaked — through SQL injection, an insider threat, a misconfigured backup, or a cloud bucket left open — every user's password is exposed instantly.

The real danger is not just your application. Users reuse passwords. A leaked password from your service can be used in **credential stuffing** attacks against:

- Email accounts
- Banking portals
- Social media
- Government services

One leak can cascade into thousands of compromised accounts across the web.

---

## 2. Encryption vs. Hashing — Know the Difference

A common interview mistake is saying "we **encrypt** the passwords."

| | Encryption | Hashing |
|---|---|---|
| Reversible? | Yes (with the key) | No (one-way) |
| Needs key management? | Yes | No |
| Right tool for passwords? | No | **Yes** |

### Why not encryption?

- Encryption is **reversible**. If an attacker steals the key, every password is exposed.
- Managing encryption keys (key rotation, key chains, HSMs) is expensive and complex.
- You do not need to recover the original password — you only need to **verify** it. Hashing fits this perfectly.

### Why hashing works

Hashing is a one-way function. Given the hash, you cannot mathematically reverse it back to the original input. To verify a login, you hash the submitted password and compare it to the stored hash.

---

## 3. Why Hashing Alone is Not Enough — Enter Salt

If two users have the same password, naive hashing produces the same hash. Attackers exploit this with **precomputed tables** (dictionary tables and rainbow tables).

### What is a salt?

A **salt** is a unique, random string added to each password *before* hashing.

```
hash(password + salt) → stored_hash
```

### Rules for salting

1. **Random** — generated by a cryptographically secure RNG.
2. **Unique per password** — not per user, not global. Even if the same user resets to the same password, a new salt is generated.
3. **Stored alongside the hash** — salt is not secret; it just needs to be unique.

### Why it works

Even if two users pick `password123`, their stored hashes will be completely different because their salts differ. This makes precomputed tables useless — an attacker would need a separate rainbow table for every single salt.

---

## 4. Rainbow Table Attacks Explained

### Step 1: The Dictionary Table

Attackers precompute hash values for millions of common passwords:

```
"password123"  → 482c811da5d5b4bc6d497ffa98491e38
"qwerty"       → d8578edf8458ce06fbc5bb76a58c5ca4
"123456"       → e10adc3949ba59abbe56e057f20f883e
```

When they steal a database, they match stolen hashes against this table.

**Problem for attackers:** dictionary tables grow enormously fast. Storing every possible password's hash is impractical.

### Step 2: The Rainbow Table (Optimized)

A rainbow table compresses dictionary tables using **hash chains**:

- `H` function: plaintext → hash
- `R` function: hash → plaintext (a reduction, not a true reverse)
- Each row stores a chain: `plaintext → H → R → H → R → ...`
- Only the **start** and **end** of each chain are stored

To crack a hash, the attacker walks the chains until a match is found, then re-derives the plaintext.

### How salt defeats it

A rainbow table is built for a *specific* hashing scheme. With unique salts, the attacker would need a different rainbow table for every user — making the attack economically infeasible.

## 5. Choosing the Right Hashing Algorithm

Not all hashing algorithms are made for passwords. Here is how to think about it:

### Use case → Algorithm

| Use Case | Recommended Algorithm | Why |
|---|---|---|
| **Password storage** | **Argon2**, bcrypt, scrypt, PBKDF2 | Intentionally slow, memory-hard |
| **Data integrity (file fingerprint)** | SHA-256, SHA-3 | Strong collision resistance |
| **Fast checksum (network transfer)** | CRC32 | Fast, not security-critical |
| **Digital signatures / JWT** | HMAC-SHA-256, SHA-3 | Strong, standardized |

### What NOT to use for passwords

- **MD5** — broken since 2004. Collisions can be generated trivially.
- **SHA-1** — broken since 2005, publicly demonstrated by Google in 2017 (the "SHAttered" attack).
- **SHA-256 / SHA-3 alone** — too **fast**. They are designed for integrity, not password storage. A GPU can compute billions of SHA-256 hashes per second.

### Why "fast" is bad for passwords

If your hash function is fast, an attacker with a GPU farm can brute-force billions of guesses per second. Password hashes need to be **deliberately slow**.

---

## 6. KDFs — Key Derivation Functions

A **KDF** is a hashing function specifically designed for password storage and key derivation. Examples:

- **Argon2** — Winner of the 2015 Password Hashing Competition. The modern gold standard.
- **bcrypt** — Battle-tested since 1999. Still widely used.
- **scrypt** — Memory-hard, used by some cryptocurrencies.
- **PBKDF2** — Older but FIPS-approved. Acceptable when compliance demands it.

### What makes KDFs special

1. **Iteration count (work factor)** — The same hash is computed thousands of times. What takes a user 100ms takes an attacker 100ms × billions of guesses.
2. **Memory hardness** (Argon2, scrypt) — Requires significant RAM, defeating cheap GPU/ASIC parallelism.
3. **Tunable difficulty** — As hardware gets faster, you can increase the work factor.

### Recommended Argon2 parameters (2026)

```
Argon2id
Memory: 64 MB or higher
Iterations: 3
Parallelism: 4
Salt: 16 bytes (random)
Hash output: 32 bytes
```

## 7. Putting It All Together — A Reference Implementation

### Node.js example with Argon2

```javascript
const argon2 = require('argon2');

// Registering a user
async function hashPassword(plaintextPassword) {
  return await argon2.hash(plaintextPassword, {
    type: argon2.argon2id,
    memoryCost: 65536,    // 64 MB
    timeCost: 3,
    parallelism: 4,
  });
  // The returned string contains: algorithm + parameters + salt + hash
}

// Verifying a login
async function verifyPassword(storedHash, submittedPassword) {
  return await argon2.verify(storedHash, submittedPassword);
}
```

### Database schema

```sql
CREATE TABLE users (
  id          SERIAL PRIMARY KEY,
  email       VARCHAR(255) UNIQUE NOT NULL,
  password    VARCHAR(255) NOT NULL,  -- stores the full Argon2 string
  created_at  TIMESTAMP DEFAULT NOW()
);
```

Notice: no separate `salt` column is needed — Argon2 embeds the salt and parameters in the hash string itself.

## 8. Common Mistakes to Avoid

1. **Rolling your own crypto.** Never. Use a vetted library.
2. **Using MD5 or SHA-1 for passwords.** They are broken and far too fast.
3. **Using plain SHA-256 / SHA-3.** Fast hashes are not safe for passwords.
4. **Reusing salts across users.** Every password gets a fresh, random salt.
5. **Hardcoding pepper without rotation strategy.** A "pepper" (an app-wide secret) can help, but only with a clear rotation plan.
6. **Forgetting to upgrade work factors.** Revisit your KDF parameters every 1-2 years.
7. **Logging plaintext passwords.** Check your logging, error tracing, and analytics pipelines.
8. **Sending plaintext over HTTP.** Always use HTTPS/TLS for password transmission.

## 9. Quick Interview Cheat Sheet

| Question | Short Answer |
|---|---|
| Encrypt or hash passwords? | **Hash** — one-way, no key management. |
| Which algorithm? | **Argon2id** (or bcrypt/scrypt as fallback). |
| Why not SHA-256? | Too fast. Designed for integrity, not passwords. |
| Why MD5 is banned? | Broken collision resistance since 2004. |
| Why use salt? | Defeats rainbow tables; unique per password. |
| Where to store salt? | Alongside the hash — it does not need to be secret. |
| What is a rainbow table? | Precomputed hash chains used to crack unsalted hashes. |
| What is a KDF? | A deliberately slow, memory-hard hashing function for passwords. |

---

## 10. Final Thoughts

Storing passwords correctly is not glamorous, but it is one of the most important security decisions in any application. A single mistake here can cascade into a breach that affects millions of users across many services.

The TL;DR for 2026:

> **Use Argon2id with a unique random salt per password. Never use MD5, SHA-1, or plain SHA-256 for passwords. Transmit over HTTPS. Audit your logs.**

Get this right, and you have eliminated one of the most common — and most damaging — vulnerabilities in modern web applications.
